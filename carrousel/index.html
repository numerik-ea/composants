Carrousel

Exigences

  • Toutes les fonctionnalités, y compris la navigation entre les éléments du slideshow, doivent pouvoir être utilisées au moyen d'un clavier.
  • Les modifications apportées aux éléments du slideshow doivent être communiquées à tous les utilisateurs, y compris les utilisateurs de lecteurs d'écran.
  • Le focus clavier est géré de manière compréhensible et doit toujours être visible.
  • Les éléments qui ne sont pas actuellement affichés ne doivent pas recevoir le focus clavier, même s'ils contiennent des éléments focalisables.

Contenu

Le slideshow peut afficher un seul élément à la fois, deux (ou plus) à la fois, ou encore un seul élément sur mobile et plusieurs éléments sur des écrans plus grands (responsive).
Pour anticiper ces différentes options, il est préférable de contenir chaque élément dans un <li> et tous les éléments dans un <ul>.

  • Cas 1 : Texte uniquement (pas d'élément interactif)
    • Lorsque les utilisateurs activent les boutons pour naviguer entre les éléments, le focus est déplacé vers le conteneur de l'élément (ajouter un attribut tabindex="-1" sur le conteneur de l'élément avant de déplacer le focus vers lui).
  • Cas 2 : Uniquement un élément interactif (soit un lien, une image de lien, un bouton, un champ de saisie…)
    • Lorsque les utilisateurs activent les boutons pour naviguer entre les éléments, le focus est déplacé vers l'élément interactif (lien, bouton, champ de saisie) à l'intérieur de l'élément.
  • Cas 3 : Un élément interactif suivi d'un autre contenu (texte, image, etc.)
    • Lorsque les utilisateurs activent les boutons pour naviguer entre les éléments, le focus est déplacé vers l'élément interactif (lien, bouton, champ de saisie) à l'intérieur de l'élément.
  • Cas 4 : un contenu (texte, image, etc.) suivi d'un élément interactif
    • Lorsque les utilisateurs activent les boutons pour naviguer entre les éléments, le focus est déplacé vers le conteneur de l'élément (ajouter un attribut tabindex="-1" sur le conteneur de l'élément avant d'y déplacer le focus).
  • Cas 5 : Plusieurs éléments sont visibles en même temps
    • Lorsque le bouton Suivant est activé, le focus est déplacé vers le premier élément nouvellement visible.
    • Lorsque le bouton Précédent est activé, le focus est déplacé vers le premier élément visible du slideshow (dans l'ordre du DOM).

Le slideshow ne doit pas être en lecture automatique

Veuillez noter que nous recommandons d'éviter à tout prix d'imposer aux utilisateurs des animations non demandées. Certains utilisateurs souffrant de déficiences visuelles et/ou cognitives peuvent les trouver si dérangeantes qu'elles les amènent à quitter la page.

Si le slideshow est en lecture automatique

Les utilisateurs doivent pouvoir arrêter l'animation. Un bouton Pause/Play doit alors être le premier élément sur lequel on peut se concentrer lorsqu'on entre dans le composant avec une tabulation au clavier. Il doit également être clairement visible, présenter un rapport de contraste suffisant avec la (les) couleur(s) environnante(s) à tout moment et avoir un contour de mise au point clairement visible.

Le texte du bouton (visible ou caché s'il n'y a qu'une icône) doit indiquer aux utilisateurs ce qui se passera s'ils l'activent :

  • "Pause" pendant que le slideshow tourne,
  • "Lecture" lorsque le slideshow ne tourne pas.
  • Lorsque les utilisateurs activent le bouton, le focus doit rester sur celui-ci et le texte doit être modifié de manière dynamique.
  • Lorsque l'animation du slideshow est activée et que le curseur du clavier se déplace sur l'un des éléments visibles (éventuellement lorsque le curseur de la souris survole également l'élément), l'animation du slideshow doit s'arrêter (et l'état du bouton doit être modifié en conséquence).
  • Lorsque le curseur quitte l'élément du slideshow, l'animation peut reprendre (et l'état du bouton est à nouveau modifié).

Option : boucle

Si les utilisateurs peuvent parcourir les éléments en boucle dans les deux sens (lorsqu'ils activent le contrôle Précédent lorsqu'ils se trouvent sur le premier élément ou le contrôle Suivant lorsqu'ils se trouvent sur le dernier élément), alors.. :

sur le premier élément :

  • les boutons Précédent et Suivant sont visibles et ne sont pas désactivés,
  • le bouton Précédent porte un nom approprié. Le bouton Précédent porte un nom approprié, par exemple Précédent (aller au dernier élément),
  • lorsque les utilisateurs activent le bouton Précédent, le focus doit aller sur le dernier élément du slideshow,
  • lorsque les utilisateurs activent le bouton Suivant, le curseur doit être placé sur l'élément suivant du slideshow.

sur le dernier élément :

  • les boutons Précédent et Suivant sont visibles et ne sont pas désactivés,
  • le bouton Suivant porte un nom approprié. Par exemple, Suivant (aller au premier élément),
  • lorsque les utilisateurs activent le bouton Suivant, le focus doit aller sur le premier élément,
  • lorsque les utilisateurs activent le bouton Précédent, l'attention doit se porter sur l'élément précédent.

Option : pas de boucle

Si les utilisateurs ne peuvent pas parcourir les éléments en boucle (lorsqu'ils activent le bouton Précédent lorsqu'ils se trouvent sur le premier élément ou le bouton Suivant lorsqu'ils se trouvent sur le dernier élément), alors.. :

sur le premier élément :

  • le bouton Suivant est visible et n'est pas désactivé,
  • lorsque les utilisateurs activent le bouton Suivant, le curseur doit être placé sur l'élément suivant,
  • le bouton Précédent est visuellement désactivé (avec CSS, pas d'attribut HTML hidden) et possède un attribut aria-disabled="true",
  • lorsque les utilisateurs tentent d'activer le bouton Précédent, rien ne se passe, le focus reste sur ce bouton et l'élément affiché ne bouge pas.

sur le dernier élément :

  • le bouton Précédent est visible et n'est pas désactivé,
  • lorsque les utilisateurs activent le bouton Précédent, le focus doit aller à l'élément précédent,
  • le bouton Suivant est visuellement désactivé (avec CSS, pas d'attribut HTML hidden) et possède un attribut aria-disabled="true",
  • lorsque les utilisateurs essaient d'activer le bouton Suivant, rien ne se passe, le focus reste sur le bouton et l'élément affiché ne bouge pas.

Boutons Suivant et Précédent

< et > sont : <button type="button">

  • sont après toutes les diapositives dans l'ordre du code source
  • à l'intérieur des <button> précédent et suivant
  • un texte caché tel que <span class="sr-only">précédent</span> ou <span class="sr-only">suivant</span>
  • L'icône de la flèche est soit <svg aria-hidden="true" focusable="false">, <img alt="">, soit une image d'arrière-plan CSS.
  • ajouter le <button type="button" aria-describedby="xxx"> ou xxx = la valeur id de l'en-tête de niveau qui précède chaque slideshow
  • lorsque l'utilisateur active le bouton < ou >, le focus est déplacé sur la <div tabindex="-1"> suivante/précédente (= la nouvelle diapositive affichée) après la fin de la transition, le cas échéant.

Bouton lecture/pause

La touche lecture/pause :

  • Il s'agit de <button type="button" aria-describedby="xxx"> où xxx = la valeur d'identification de l'en-tête de section correspondant
  • contient une icône qui est soit <svg aria-hidden="true" focusable="false">, <img alt="">, soit une image d'arrière-plan CSS
  • un texte caché tel que : <span class="sr-only">Pause</span> lorsque le bouton permet de faire une pause
  • un texte caché tel que : <span class="sr-only">Play</span> lorsque lorsque la touche permet la lecture
  • Il peut s'agir du même bouton dont l'étiquette est modifiée dynamiquement.
  • Il s'agit après tout de diapositives dans le code source
